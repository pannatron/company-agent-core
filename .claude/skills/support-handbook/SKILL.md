# Customer Support Handbook

## 1. Priority & SLA Matrix
| Priority | นิยาม | First Response | Resolve |
|---|---|---|---|
| P1 — Critical | ระบบหยุด / กระทบรายได้ / ลูกค้าโกรธมาก | ≤ 1 ชม. | ≤ 4 ชม. |
| P2 — High | ฟีเจอร์หลักใช้ไม่ได้ มีคนกระทบหลายคน | ≤ 4 ชม. | ≤ 24 ชม. |
| P3 — Normal | คำถามทั่วไป ปัญหามี workaround | ≤ 8 ชม. | ≤ 48 ชม. |
| P4 — Low | feature request, การสะกด, ความเห็น | ≤ 24 ชม. | ≤ 7 วัน |

## 2. Ticket Status Flow
```
open → in_progress → waiting_customer ⇄ in_progress → resolved → closed
                                                 ↘ reopened ↗
```

## 3. Tone Guide
- ใช้ชื่อจริงลูกค้า (ถ้ามี)
- **Empathy first** — รับฟังก่อนเสนอวิธีแก้
- ใช้ "เรา" ไม่ใช่ "ทางบริษัท"
- ไม่ใช้คำเทคนิคที่ลูกค้าไม่ได้พิมพ์ใช้ก่อน
- **ห้าม** ใช้คำว่า "นโยบายเรา / ทำไม่ได้" โดยไม่เสนอทางออก

## 4. เทมเพลตการตอบ (4 รูปแบบ)

**a) Acknowledge (ทันทีที่รับ)**
```
สวัสดีคุณ <ชื่อ>

ขอบคุณที่แจ้งเข้ามาค่ะ/ครับ เรารับเรื่อง <สรุปปัญหา> เรียบร้อย
ทีมเรากำลังตรวจสอบ จะอัปเดตกลับภายใน <SLA> นะคะ/ครับ

ขอบคุณที่ติดตามเรา
```

**b) Clarify (ต้องการข้อมูลเพิ่ม)**
```
เพื่อช่วยได้ตรงจุด ขอข้อมูลเพิ่มอีก 2-3 ข้อค่ะ/ครับ:
1. <คำถามเฉพาะ>
2. ...
3. screenshot หรือ error code (ถ้ามี)

ระหว่างนี้แนะนำลอง <workaround ถ้ามี> ดูก่อนนะคะ/ครับ
```

**c) Resolve**
```
อัปเดตค่ะ/ครับ — <สาเหตุ + วิธีแก้สั้นๆ>
ตอนนี้ <สถานะ> แล้ว ลองตรวจสอบดูอีกครั้งได้เลย

ถ้ายังมีปัญหา ตอบกลับเมลนี้ได้ทันที เรายังเปิด ticket ไว้ 48 ชม.
ขอบคุณที่ให้โอกาสได้แก้ไขนะคะ/ครับ
```

**d) Apology (เกิด incident หรือ delay)**
```
ขอโทษจริงๆ ค่ะ/ครับ ที่ <ระบุสิ่งที่พลาด> ไม่ใช่ประสบการณ์ที่เราตั้งใจให้คุณได้รับ

เราได้ <action ที่ทำเพื่อแก้> และจะ <action ป้องกันไม่ให้เกิดซ้ำ>
เพื่อชดเชย เราขอ <compensation ถ้ามี>

หากมีอะไรเพิ่มเติม บอกได้ทันทีค่ะ/ครับ
```

```markdown
**Updated:** YYYY-MM-DD

## อาการ
...

## วิธีแก้
1. ...
2. ...

## ถ้ายังไม่ได้
ติดต่อ support ที่ ... (priority: P?)
```

## 6. เมื่อไหร่ต้องเพิ่ม KB
- คำถามเดิม ≥ 3 ครั้งใน 30 วัน
- ปัญหาที่ใช้เวลา resolve > 2 ชั่วโมงและจะเกิดซ้ำได้
- หลัง incident → post-mortem สั้น
